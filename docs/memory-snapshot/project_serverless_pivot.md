**Major architecture change, committed `17ab10c` (Step 1.15 mid-flight).** The deployment target moved from **ECS Fargate → AWS Lambda + Aurora Serverless v2 Data API**. SPEC.md §2.2/§2.4/§3.3.3/§3.3.5/§4 and CLAUDE.md were rewritten to match. Decided with the user after a cost review.

**The driving constraint:** VPC interface endpoints only reach *AWS* services, but the scan task's heaviest egress is to *non-AWS* hosts (Binance/Anthropic/Telegram). In a VPC that needs a NAT (~$32/mo) or a public IP. **Lambda outside any VPC** gets free internet egress; the DB is reached over the **RDS Data API (HTTPS, IAM-auth)**, so no VPC attachment. A minimal VPC hosts only Aurora in isolated subnets.

**Confirmed decisions:**
1. Architecture: Lambda (no VPC, container image via ECR) + Aurora Serverless v2 (pgvector, **scale-to-zero**) + RDS Data API + EventBridge Scheduler → Lambda. No NAT, no interface endpoints, no ECS.
2. Persistence: **dual backend** behind one repository interface — `AsyncpgRepository` (local Docker dev, unchanged) + `DataApiRepository` (boto3 `rds-data`, cloud). Config switch `PERSISTENCE_BACKEND=asyncpg|dataapi`. This preserves the working local dev loop + integration tests.
3. Dashboard (Slice 4) is the ONE non-Lambda piece — long-running Fargate/App Runner service (WebSocket + LISTEN/NOTIFY need a persistent connection).
